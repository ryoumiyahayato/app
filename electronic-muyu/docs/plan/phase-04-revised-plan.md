## 一、三种后台能力等级

| 等级 | 名称 | 说明 | 实现方式 | 状态 |
|------|------|------|----------|------|
| 等级 1 | 前台可用 | App 在前台时 WSS 实时收发 tap | 阶段 3 已完成（Activity + ViewModel + OkHttp WebSocket） | ✅ 已完成 |
| 等级 2 | 短期后台可用 | App 切后台后，进程未被系统杀死时尽量维持连接；不承诺长期后台、不承诺熄屏、不承诺省电后可用 | 自然维持：Activity onStop 时不主动断连，系统若未杀进程可继续收发；**无 Foreground Service** | ⬜ 本次待做 |
| 等级 3 | 长期后台尽量在线 | 需要 Foreground Service + 常驻通知；需要 service type 选择；需面对 Android 15+ 限制 | Foreground Service + WSS | ⬜ 后续单独评估 |

### 等级 2 的能力边界

等级 2 能做和不能做的：

| 场景 | 等级 2 预期 |
|------|------------|
| App 切后台，10 秒内 | ✅ 大概率仍可收发 |
| App 切后台，30 秒内 | ⚠️ 视系统版本/厂商而定 |
| App 切后台，1-5 分钟 | ❌ 大概率断连 |
| 熄屏 | ❌ 断连概率高 |
| 省电模式 | ❌ 进程被杀 |
| 用户划掉 App | ❌ 进程被杀 |
| 切后台后断连，回到前台时 | ✅ 自动重连（阶段 3 已有的退避策略） |

**等级 2 的本质**：不是"后台提醒方案"，而是"切后台短时间内如果系统没杀进程，尽量不丢连接"。它是一种尽力而为但不承诺的策略。

## 二、foregroundServiceType 重新评估

### 2.1 dataSync

| 项目 | 内容 |
|------|------|
| 官方定义 | 数据同步、上传、下载等一次性或有限时长的数据传输 |
| 对本项目的表面匹配度 | 中等——WSS 本质是双向数据通道，可解释为"tap 事件同步" |
| ✅ 优点 | 实现简单，不需要额外审核，Manifest 直接声明即可 |
| ❌ **关键风险** | **Android 15（API 35）新增限制：dataSync 类型的 Foreground Service 在 24 小时内累计运行不得超过 6 小时。超时后系统会回调 `Service.onTimeout(int, int)`，App 必须处理** |
| ❌ 对电子木鱼的影响 | 电子木鱼期望的是**长期在线**（全天候待机），6 小时限制意味着用户每天至少有 18 小时无法通过 dataSync 保持后台连接。这不是"偶发断开"，而是系统强制下线 |
| ❌ 其他风险 | Google Play 可能认为双人提醒不属于 dataSync 的合理用途 |
| **结论** | **不能作为长期 WSS 常驻的默认方案。** 只能作为有限时长（如会议模式、临时监听）的候选，不适合全天候后台连接 |

### 2.2 remoteMessaging

| 项目 | 内容 |
|------|------|
| 官方定义 | 远程消息通信，如聊天 App 的实时消息接收 |
| 对本项目的表面匹配度 | 低——电子木鱼不是聊天工具，没有自由文本消息 |
| ✅ 优点 | 如果用 remoteMessaging，系统可能给予更宽松的后台限制 |
| ❌ 风险 | **滥用风险高**：Google Play 和系统会检查 App 是否真的在做"远程消息通信"。电子木鱼既不显示消息列表、不保存消息、不提供聊天 UI，一旦被判定为滥用 remoteMessaging 类型，可能被拒绝上架或系统强制降级 |
| **结论** | **不作为候选。** 本项目没有远程消息需要"接收"，只有一个二进制的 tap 信号 |

### 2.3 specialUse

| 项目 | 内容 |
|------|------|
| 官方定义 | 系统集成、设备管理、企业应用等特殊用途 |
| 对本项目的表面匹配度 | 低——电子木鱼只是一个双人提醒工具 |
| ✅ 优点 | 无 6 小时累积限制 |
| ❌ **关键风险** | **Google Play 审核门槛高**：需要提交详细的使用说明，解释为什么不能使用其他 service type。电子木鱼没有充分的理由 |
| ❌ **次要风险** | 国内厂商系统对 specialUse 同样有限制，不一定优于 dataSync |
| **结论** | **不作为第一版默认实现。** 作为后续候选方案需要更充分的论证 |

### 2.4 不使用 Foreground Service

| 项目 | 内容 |
|------|------|
| 能力等级 | 等级 1（前台）+ 等级 2（短期后台尽力维持） |
| ✅ 优点 | 无需 Foreground Service、无需常驻通知、无需面对 service type 选择、无需面对 6 小时限制、Manifest 无需新增权限、用户无感知 |
| ❌ 缺点 | 后台收发能力弱：切后台后只能依靠系统未杀进程的短暂窗口期（通常 10 秒到几分钟不等） |
| **结论** | **可以作为第一版保守方案。** 如果用户接受"前台实时 + 后台不一定收到"的定位，可以不做 Foreground Service |

## 三、推荐路线

### 路线 A：保守第一版（推荐先做）

**不做 Foreground Service，不做后台常驻。**

| 项目 | 内容 |
|------|------|
| 后台能力 | 等级 1（前台）+ 等级 2（短期后台尽力维持） |
| Foreground Service | ❌ 不做 |
| 常驻通知 | ❌ 不做 |
| service type | ❌ 不需要 |
| POST_NOTIFICATIONS | ⚠️ 阶段 4A 做，只做前台通知能力 |
| 优点 | 权限最少、边界最清楚、无 Android 15 限制、无审核风险、实现简单 |
| 缺点 | 用户切后台后，仅系统未杀进程的短暂窗口期可收发 |

### 路线 B：有限后台版（阶段 4B 评估）

**使用 Foreground Service，但需单独评估。**

| 项目 | 内容 |
|------|------|
| 后台能力 | 等级 3（长期后台尽量在线） |
| Foreground Service | ✅ 需要 |
| service type | 待定（dataSync 有 6 小时限制，specialUse 审核门槛高） |
| 常驻通知 | ✅ 需要 |
| Android 15 6 小时限制 | 必须处理 onTimeout |
| 决策点 | **是否接受 6 小时限制？是否接受常驻通知对用户的干扰？是否愿意承担 specialUse 审核风险？** |

### 建议策略

**第一版先走路线 A（阶段 4A），后续再评估路线 B（阶段 4B）。**

理由：
1. 阶段 4A 的增益是确定的：App 进程存活时收到 tap 可发通知栏提醒。
2. 阶段 4B 的收益和成本需要实际测试后才能判断：不同厂商系统对 Foreground Service 的实际限制差异太大，先做 4A 再测试，收集数据后再决定是否走 4B。
3. 阶段 4B 的真实用户体验可能并不比 4A 好多少（厂商系统可能杀掉 Foreground Service），不值得前期投入大量实现成本。

## 五、关键问题解答

### Q1: 为什么 dataSync 不能作为长期 WSS 常驻的默认方案？

**答**：Android 15（API 35）引入了 `Service.onTimeout(int, int)` 回调，对 `dataSync` 类型的 Foreground Service 施加了**24 小时内累计不超过 6 小时**的运行限制。超时后系统会强制停止 Service。

这意味着：
- 即使用户一直开着 App 在后台，每天最多 6 小时能通过 dataSync 保持后台连接
- 剩余 18 小时，后台连接能力降级为等级 2（进程侥幸存活时可用）
- 这不是"偶发不稳定"，而是系统级的强制离线

电子木鱼的目标是"全天候后台在线提醒"，dataSync 从设计上就不适合这个场景。

### Q2: Android 15 的 6 小时限制对本项目有什么影响？

**答**：
- 如果走 dataSync，每天有 18 小时无法保证后台连接
- 用户可能在不同时段使用，6 小时窗口不可预测
- 需要实现 `onTimeout` 回调，超时后的策略需要明确（静默降级？通知用户？）
- 对于双人提醒场景，6 小时窗口意味着"白天能用，晚上可能不行"，或者相反
- 对于一对特定用户来说，如果他们在同一时间段使用（比如上班时间），6 小时可能刚好够用
- 但无法向用户承诺"任何时候都能收到"

### Q3: specialUse 的审核风险是什么？

**答**：
- Google Play 要求提交详细的用途说明，解释为什么不能使用标准 service type
- 审核周期不确定，可能被拒绝
- 本项目不走 Play 商店发布（独立安装），所以 Play 审核本身不是问题
- 但国内厂商系统对 `specialUse` 类型的 Foreground Service 同样有额外限制，不一定比 `dataSync` 好
- 如果后续要走 Play 商店发布，`specialUse` 审核可能成为障碍
- **结论**：不走 Play 商店时可以作为候选，但实现成本和技术风险高于 `dataSync`

### Q4: 是否建议第一版先做 4A，而不是直接做 Foreground Service？

**答**：**是的，强烈建议先做 4A。**

理由：
1. 阶段 4A 的收益是确定的——App 存活时收到 tap 有通知栏提醒。
2. 阶段 4B 只能解决"App 存活"的问题，无法解决"App 被杀"的问题，而后者才是用户最关心的。
3. 厂商系统差异巨大，FGS 的实际效果需要真机测试后才能评估。
4. 先做 4A 后做 4B，可以在 4A 完成后实际测试后台存活情况，收集数据后再决定是否值得做 FGS。
5. 避免在"无法确定是否有效"的方案上投入过多实现成本。

### Q5: 如果做 4A，后台能力边界是什么？

**答**：
```
App 前台       → ✅ 实时收到，通知栏提醒（如有权限）
切后台 < 30秒  → ⚠️ 大概率收到（系统未杀进程时）
切后台 > 1分钟  → ❌ 大概率收不到（系统限制后台网络）
熄屏           → ❌ 大概率收不到
省电模式       → ❌ 收不到
用户划掉       → ❌ 收不到
用户强停       → ❌ 收不到
切后台后回到前台 → ✅ 自动重连，恢复收发（阶段 3 已有）
```

### Q7: 如果用户关闭通知权限，会发生什么？

**阶段 4A**（无 FGS）：
- WSS 连接不受影响（仍在主进程运行）
- App 前台时，UI 提醒正常（收到提醒次数 +1、显示提示、音效、震动）
- 通知栏无法发出提醒
- 设置页应显示"通知未开启"

**阶段 4B**（有 FGS）：
- FGS 常驻通知无法显示 → 有些系统会因此杀掉 FGS
- 即使 WSS 仍在运行，用户收不到通知栏提醒
- 通知权限关闭后，Foreground Service 的存活本身也受影响

### Q8: 如果 App 被杀、强停、省电限制，会发生什么？

| 场景 | 阶段 4A | 阶段 4B |
|------|---------|---------|
| 用户划掉 App（多任务删除） | ❌ 进程杀死，无法收发 | ❌ 进程杀死，无法收发（FGS 也被杀） |
| 用户强停（系统设置） | ❌ 进程杀死 | ❌ 进程杀死 |
| 省电模式 | ❌ 进程可能被杀 | ❌ FGS 也可能被杀 |
| 系统内存不足回收 | ❌ 进程被杀 | ⚠️ FGS 被杀概率低于普通进程 |
| 熄屏后系统限制 | ❌ 后台网络可能被切断 | ⚠️ FGS 可维持网络，但厂商差异大 |

**核心事实**：4B 的增量优势是"进程存活窗口更长"，但无法解决"进程被杀"的问题。厂商系统在省电模式下可能连 FGS 一起杀。

## 六、阶段 4A 详细设计

### 6.1 POST_NOTIFICATIONS 权限申请

**触发时机**：首次启动 App 时，如果 SDK >= 33，自动弹窗申请。

**流程**：
1. `MainActivity.onCreate` 中检查 `Build.VERSION.SDK_INT >= 33`
2. 如果已授权，直接创建通知渠道
3. 如果未授权，调用 `requestPermissions(arrayOf(POST_NOTIFICATIONS), REQUEST_CODE)`
4. 用户拒绝后不重复弹窗，在设置页显示"通知未开启，前往系统设置开启"
5. 设置页提供"前往设置"按钮 → `Intent(Settings.ACTION_APP_NOTIFICATION_SETTINGS)`

**权限状态持久化**：
- 不存到 DataStore，每次动态检查
- 设置页使用 `checkSelfPermission` 实时读取

### 6.2 通知渠道

```
Channel ID:   muyu_tap_channel
Channel 名称: 功德提醒
重要性:       IMPORTANCE_HIGH（弹窗、震动、提示音）
描述:         收到对方提醒时通知
```

创建时机：`MainActivity.onCreate` 中，每次启动时调用 `createNotificationChannel`（幂等操作）。

### 6.3 通知提醒

**触发时机**：`WebSocketClient` 收到 tap → `MainViewModel.onReceiveTap()` → 调用 `NotificationHelper.showTapNotification(context)`

**通知内容**：
```
标题: "电子木鱼"
正文: "收到一次功德提醒"
```

**通知 ID**：固定 `1001`，每次更新同一通知。

**PendingIntent**：
```kotlin
val intent = Intent(context, MainActivity::class.java).apply {
    flags = FLAG_ACTIVITY_NEW_TASK or FLAG_ACTIVITY_CLEAR_TOP
}
val pendingIntent = PendingIntent.getActivity(
    context, 0, intent,
    FLAG_UPDATE_CURRENT or FLAG_IMMUTABLE
)
```

**提醒效果**：
- 仅显示通知栏通知
- 不额外播放自定义音效（使用系统默认通知音）
- 不额外震动（使用系统默认震动）
- 如果 App 在前台，可以选择不发通知（因为 UI 已有提醒），也可以选择同时发通知
  - **建议**：前台也发通知，因为通知可以下拉查看，且不影响 UI 体验

### 6.4 前台 vs 后台区分

**简单方案**：收 tap 时直接发通知，不区分前后台。

理由：
1. 前台发通知不影响体验（通知只是多一个下拉渠道）
2. 不需要引入 Activity 生命周期检测的复杂度
3. 不需要在 Service 和 Activity 之间做状态通信
4. 一直发通知，用户下拉通知栏可以看到历史提醒条数

如果后续需要区分前后台，可以这样实现：
```kotlin
// 在 MainViewModel 中维护前台状态
var isAppInForeground = mutableStateOf(false)

// MainActivity.onResume → true
// MainActivity.onPause → false

// 收到 tap 时：
if (isAppInForeground) {
    // UI 更新 + 可选发通知
} else {
    // 发通知（后台上才会看到通知）
}
```

阶段 4A 采用简单方案：**统一发通知，不区分前后台**。

### 6.5 设置页通知权限显示

新增内容：
